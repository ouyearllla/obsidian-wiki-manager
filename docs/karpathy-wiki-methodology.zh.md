# LLM Wiki —— 一种使用 LLM 构建个人知识库的模式

> 作者：[Andrej Karpathy](https://karpathy.ai)

一种使用 LLM 构建个人知识库的模式。

这是一份理念文件，设计用于复制粘贴到你自己的 LLM 智能体中（如 OpenAI Codex、Claude Code、OpenCode / Pi 等）。它的目标是传达高层理念，具体细节将由你的智能体与你协作构建。

## 核心理念

大多数人使用 LLM 处理文档的体验都类似于 RAG：你上传一组文件，LLM 在查询时检索相关片段，然后生成答案。这能用，但 LLM 每次都在从零重新发现知识。没有积累。问一个需要综合五篇文档的微妙问题，LLM 每次都得重新找到并拼凑相关片段。什么都没有被构建起来。NotebookLM、ChatGPT 文件上传和大多数 RAG 系统都是这样工作的。

这里的理念不同。LLM 不只是在查询时从原始文档中检索，而是**增量构建并维护一个持久化的 wiki** —— 一个位于你和原始资料之间的、结构化的、相互关联的 Markdown 文件集合。当你添加一份新资料时，LLM 不只是为以后的检索建立索引。它会阅读、提取关键信息，并将其整合到现有 wiki 中 —— 更新实体页面、修订主题摘要、标注新数据与旧结论矛盾的地方、强化或挑战不断演进的综合分析。知识被编译一次，然后*持续保持更新*，而不是在每次查询时重新推导。

这就是关键区别：**wiki 是一个持续存在的、不断复利的产物。** 交叉引用已经就位。矛盾已被标记。综合分析已经反映了你读过的所有内容。每增加一份资料、每提一个问题，wiki 都变得更加丰富。

你几乎从不（或很少）自己写 wiki —— LLM 负责撰写和维护全部内容。你负责资料搜集、探索和提出正确的问题。LLM 完成所有苦活 —— 摘要、交叉引用、归档和记录，这些才是让知识库随时间真正有用的东西。在实践中，我一边开着 LLM 智能体，另一边开着 Obsidian。LLM 根据我们的对话进行编辑，我实时浏览结果 —— 跟踪链接、查看图谱视图、阅读更新的页面。Obsidian 是 IDE；LLM 是程序员；wiki 是代码库。

## 应用场景

- **个人**：追踪你自己的目标、健康、心理、自我提升 —— 归档日记、文章、播客笔记，随时间构建关于自己的结构化画像。
- **研究**：在数周或数月内深入一个主题 —— 阅读论文、文章、报告，增量构建一个具有演进论点的综合 wiki。
- **读书**：边读边归档每一章，为角色、主题、情节线及其关联构建页面。
- **商业/团队**：由 LLM 维护的内部 wiki，以 Slack 对话、会议记录、项目文档、客户通话为输入。
- **竞品分析、尽职调查、旅行规划、课程笔记、兴趣深挖** —— 任何你在持续积累知识并希望它被组织而非散落各处的场景。

## 架构：三个层级

| 层级 | 内容 | 由谁维护 |
|------|------|---------|
| **原始资料层** (`raw/`) | 你精心策展的源文档集合。不可变，LLM 只读 | 你 |
| **Wiki 层** (`wiki/`) | LLM 生成的 Markdown 目录。摘要、实体、概念、对比 | LLM |
| **Schema 层** (`CLAUDE.md`) | 告诉 LLM wiki 的结构、约定和工作流 | 双方共同演进 |

## 操作

**摄取（Ingest）。** 你把一份新资料放入原始资料层，然后告诉 LLM 处理它。LLM 阅读资料，与你讨论关键要点，在 wiki 中撰写摘要页面，更新索引，更新相关的实体和概念页面，并在日志中追加记录。一份资料可能触及 10-15 个 wiki 页面。

**查询（Query）。** 你针对 wiki 提问。LLM 搜索相关页面，阅读它们，并综合出带引用的答案。重要洞察：**好的答案可以作为新页面归档回 wiki。** 你要求的对比分析、发现的关联 —— 这些都是有价值的，不应消失在聊天记录中。

**健康检查（Lint）。** 定期让 LLM 对 wiki 进行健康检查。查找：页面之间的矛盾、已被新资料取代的过时结论、没有入链的孤儿页面、被提到但缺少独立页面的重要概念、缺失的交叉引用。这能让 wiki 在成长过程中保持健康。

## 索引与日志

- **index.md** 是面向内容的。它是 wiki 中所有内容的目录。LLM 在每次摄取后更新它。回答查询时，LLM 先读取索引找到相关页面，然后深入阅读。
- **log.md** 是面向时间的。它是一个仅追加的记录。如果每条记录以一致的前缀开头（如 `## [YYYY-MM-DD] 摄取 | 文章标题`），日志就变得可以用简单的 Unix 工具解析。

## 为什么这行得通

维护知识库的苦活不是阅读或思考 —— 而是簿记工作。更新交叉引用、保持摘要最新、标注新数据与旧结论的矛盾、在数十个页面间维持一致性。人类放弃 wiki 是因为维护负担的增长速度快于价值的增长。LLM 不会感到无聊，不会忘记更新交叉引用，能在一次操作中触及 15 个文件。Wiki 保持被维护的状态，因为维护成本接近于零。

人类的工作是策展资料、指导分析、提出好问题、思考这一切意味着什么。LLM 的工作是其他所有事情。

## 与 Memex 的关系

这个理念在精神上与 Vannevar Bush 的 Memex（1945）相关 —— 一个个人的、精心策展的知识存储，文档之间有联想式路径。Bush 的愿景与其说接近今天的互联网，不如说更接近这个：私人的、主动策展的、文档之间的连接与文档本身一样有价值。他无法解决的部分是"谁来做维护"。LLM 解决了这个问题。

## 附注

这份文档有意保持抽象。它描述的是理念，而非具体实现。确切的目录结构、schema 约定、页面格式、工具链 —— 这一切都取决于你的领域、你的偏好和你选择的 LLM。上面提到的所有内容都是可选和模块化的 —— 挑有用的用，忽略不适用的。正确的使用方式是将这份文档分享给你的 LLM 智能体，然后一起协作实例化一个适合你需求的版本。
